# Flutter(Riverpod) 진료실 데이터 흐름 구조 — 처음 보는 사람을 위한 긴 설명

> 한 줄 요약: "화면이 데이터를 받기까지" 거치는 계층(라우트 → Provider → Service → Mock/실서버 → 모델)을
> mediclo 진료실 예시로 끝까지 따라가며, Riverpod·freezed·Either·값객체가 각자 무슨 일을 하는지 설명한다.

이 노트는 발단이 된 실제 작업 기록과 짝이다 → [[2026-06-06-consultation-patient-memo-통합]]
대상 독자는 **Flutter/Riverpod 가 거의 처음이지만 이 코드베이스 구조를 이해하고 싶은 사람**이다.

## 0. 큰 그림 — 데이터는 이 길을 따라 흐른다

```
[사용자가 차트 탭]
      │  scheduleId(예약번호) 하나만 넘김
      ▼
[GoRouter]  /consultation/:scheduleId
      ▼
[ConsultationPage] ── 각 탭 위젯에 scheduleId 전달
      ▼
[탭 위젯]  scheduleId → customerId 로 변환 (현황판에서 lookup)
      ▼
[Provider]  ref.watch(xxxProvider(customerId))      ← Riverpod
      ▼
[Service]  fetch/create/update/delete                ← 추상 인터페이스
      │
      ├── Mock 모드면 → MockXxxService (가짜 데이터)
      └── 아니면     → XxxServiceImpl (실서버, Dio HTTP)
      ▼
[DTO]  서버 응답 원형(PatientMemoDto)
      ▼
[Mapper]  DTO → 화면 모델(Memo/Jusojung/Etc)로 변환
      ▼
[위젯]  화면에 그림
```

핵심 직관 하나만 가져가자: **각 계층은 "한 가지 일"만 한다.**
라우트는 길 안내, Provider 는 "이 데이터 줘"를 캐싱하며 들고 있기, Service 는 실제 가져오기,
Mapper 는 서버 모양을 화면 모양으로 번역. 이렇게 나눠두면 한 곳을 바꿔도 다른 곳이 안 흔들린다.

## 1. 라우트는 왜 `customerId`가 아니라 `scheduleId`만 넘길까

진료실 주소는 `/consultation/:scheduleId` 다. 환자ID(`customerId`)가 아니라 **예약ID(`scheduleId`)** 가 키다.
- 현실에서 "진료실에 들어간다"는 건 "이 예약 건을 본다"는 뜻이라 예약ID가 자연스러운 입구다.
- 환자 정보(이름·customerId 등)는 **현황판(status board)이 이미 들고 있다.** 그래서 진료실은
  환자 정보를 통째로 안 받고, 예약ID로 현황판에서 찾아 쓴다.

그 "찾기"를 함수 하나로 묶어둔 게 이거다:
```dart
int? resolveCustomerIdFromSchedule(List<ScheduleModel> allSchedules, String scheduleId) {
  // allSchedules 에서 scheduleId 가 같은 예약을 찾아 → 그 예약의 patient.customerId 반환
  // 못 찾으면 null
}
```
**교훈:** 같은 변환 로직을 여러 곳에서 복붙하지 말고 함수 한 곳에 둔다.
이번 버그의 본질도 "주소증 탭만 이 함수를 안 쓰고 `customerId`를 상수로 박아놨다"는 것이었다.

## 2. Riverpod Provider — "데이터를 들고 있는 상자"

Riverpod 는 상태관리 라이브러리다. 위젯이 직접 서비스를 호출하는 대신, **Provider** 라는 상자에서
데이터를 꺼내 쓴다(`ref.watch`). 상자가 알아서 캐싱·갱신·재호출을 관리한다.

이 프로젝트엔 두 가지 모양이 나온다.

### (a) `FutureProvider.family<결과, 키>` — 비동기 + 파라미터
```dart
final etcListProvider = FutureProvider.family<List<Etc>, int>((ref, customerId) async {
  final service = ref.watch(patientMemoServiceProvider);
  final result = await service.fetch(customerId: customerId, category: MemoCategory.etc);
  return result.fold((f) => throw Exception(f.message), (dtos) => dtos.map(toEtc).toList());
});
```
- `family<List<Etc>, int>` = "정수(customerId) 하나를 받아 `List<Etc>` 를 비동기로 주는 Provider".
- 위젯에서는 `ref.watch(etcListProvider(customerId))` 처럼 **키를 넣어** 호출한다.
  키가 다르면 다른 결과를 따로 캐싱한다(환자별로 분리되는 이유).
- 화면에서는 보통 `.when(loading:, error:, data:)` 로 세 상태를 그린다.

### (b) `@riverpod class ... extends _$...` — 코드 생성형 Notifier
주소증은 핀 토글·낙관적 업데이트 같은 "동작"이 있어서 클래스형을 쓴다.
```dart
@riverpod
class CurrentJusojungList extends _$CurrentJusojungList {
  @override
  Future<List<Jusojung>> build(String scheduleId) async { ... }  // 최초 로딩
  Future<void> togglePin(String id) async { ... }                // 사용자 동작
}
```
- `part 'current_jusojung_provider.g.dart';` 와 `build_runner` 가 `_$...` 부모 클래스를 자동 생성한다.
- `build(...)` 의 인자가 곧 `family` 키가 된다. 여기선 `scheduleId`.
- **중요:** `build` 안에서 `ref.watch(other)` 를 하면, other 가 바뀔 때 자동으로 다시 빌드된다.
  그래서 "Mock 모드 토글 시 자동 재조회" 같은 게 공짜로 된다.

## 3. Service 계층 — "실제로 가져오는 사람" + Mock 토글

Provider 는 데이터를 "들고만" 있고, 진짜로 서버를 때리는 건 **Service** 다.
이 프로젝트는 추상 인터페이스 + 두 구현(실서버/Mock)을 둔다.

```dart
abstract class PatientMemoService {            // 계약(인터페이스)
  Future<Either<PatientMemoFailure, List<PatientMemoDto>>> fetch({ ... });
  Future<Either<PatientMemoFailure, PatientMemoDto>> create({ ... });
  // ...
}
class PatientMemoServiceImpl implements PatientMemoService { ... }  // 실서버(Dio)
class MockPatientMemoService implements PatientMemoService { ... }  // 가짜 데이터

final patientMemoServiceProvider = Provider<PatientMemoService>((ref) {
  if (ref.watch(mockModeProvider)) return MockPatientMemoService();    // 토글로 분기
  return PatientMemoServiceImpl(ref.watch(dioProvider));
});
```

왜 이렇게? **서버가 없거나 불안정해도 UI 를 검증**하기 위해서다.
디자인/QA 가 실서버 없이 화면을 보려면 가짜 데이터가 필요하다. `mockModeProvider` 토글 하나로 둘을 바꾼다.
Provider 가 `ref.watch(mockModeProvider)` 를 보고 있어, 토글하면 위 Provider 가 다시 만들어지고 → 그걸 watch 하던 화면도 자동 갱신된다.

### `Either<Failure, T>` 가 뭐냐
fpdart 의 타입으로 "성공(Right) 아니면 실패(Left)"를 **한 값에 담는다.** 예외를 던지는 대신 결과로 돌려준다.
```dart
final result = await service.fetch(...);
return result.fold(
  (failure) => throw Exception(failure.message), // Left: 실패 처리
  (dtos)    => dtos.map(toEtc).toList(),          // Right: 성공 처리
);
```
호출측이 성공/실패 두 갈래를 **빼먹지 않고** 다루게 강제하는 게 장점이다. `failure.code` 로 401 같은 분기도 한다.

## 4. DTO vs 모델, 그리고 Mapper — "서버 모양"과 "화면 모양"은 다르다

서버가 주는 JSON 한 줄을 그대로 담은 게 **DTO** 다.
```dart
class PatientMemoDto {          // 서버 응답 원형 (대문자 키 그대로)
  final int memoId;     // MEMOID
  final int memoType;   // MEMOTYPE  (0=메모, 1=주소증, 4=기타 ...)
  final int top;        // TOP (1=핀 고정)
  final String? crtime; // CRTIME (작성시각, 정렬·N배지 계산용)
  // ...
}
```
화면에서 쓰기 좋은 모양은 따로 있다(freezed 불변 모델).
```dart
@freezed
class Etc with _$Etc {
  const factory Etc({
    required int id, required int customerId, required String content,
    required DateTime date,
    @Default(false) bool isPinned,
    @Default(false) bool isNew,        // "오늘 작성" → N 배지
    @Default(7) int colorIndex,        // 본문 색상(7=검정)
  }) = _Etc;
}
```
둘 사이를 번역하는 게 **Mapper** 다.
```dart
Etc toEtc(PatientMemoDto d) => Etc(
  id: d.memoId, customerId: d.customerId, content: d.memo,
  date: parseMemoDate(d.memoDate) ?? DateTime.now(),
  isPinned: d.top == 1,
  isNew: /* crtime 이 오늘이면 true */,
  colorIndex: int.tryParse(d.color ?? '') ?? 7,
);
```
왜 분리? 서버 필드명이 바뀌어도 **DTO·Mapper 만 고치면 되고 화면 모델은 안전**하기 때문.
freezed 는 `copyWith`·동등성·불변성을 자동 생성해줘서 모델을 안전하게 다루게 해준다(`*.freezed.dart` 가 그 산출물).

> 메모/주소증/기타는 **같은 DTO(`PatientMemoDto`)** 를 쓰되 Mapper 만 3개(`toMemo`/`_toJusojung`/`toEtc`)다.
> 같은 서버 데이터를 화면 목적에 맞게 다르게 번역하는 것.

## 5. "엔드포인트 1개면 서비스도 1개" — 기타 통합 이야기

처음엔 기타(Etc)가 자기만의 `EtcService`(+Mock)를 갖고 있었다. 그런데 백엔드를 보니
주소증·메모·기타가 **전부 같은 URL** `/v2/consultation/patient-memo` 로 처리되고,
구분은 숫자 컬럼 하나(`MEMOTYPE`: 메모0/주소증1/기타4)였다.

```dart
enum MemoCategory {
  jusojung, memo, etc;
  int get filterType => switch (this) {           // GET 조회 필터
    MemoCategory.jusojung => 0, MemoCategory.memo => 1, MemoCategory.etc => 2 };
  int get memoType => switch (this) {             // POST/PUT 저장값
    MemoCategory.jusojung => 1, MemoCategory.memo => 0, MemoCategory.etc => 4 };
}
```

그래서 기타 전용 서비스를 지우고 `patientMemoServiceProvider` 에 `category: MemoCategory.etc` 로 합쳤다.
**왜?** 서비스를 엔드포인트와 1:1 로 맞추면, 같은 URL 을 때리는 Dio 호출·에러 매핑·JSON 디코드 로직을
한 번만 쓴다. 3개로 쪼개면 그 보일러플레이트가 3벌이 된다(중복 = 버그 3배).

판단 기준(일반화): **"백엔드 리소스가 하나면 클라 서비스도 하나"** 가 기본값.
나중에 정말 스키마가 갈라지면 그때 쪼갠다(미리 쪼개는 건 과설계, YAGNI).

## 6. 값객체(Value Object)로 시그니처 오염 막기

통합하니 `create/update` 에 도메인 전용 파라미터가 평평하게 쌓였다.
```dart
// 나빠지는 방향: 주소증 호출자도 color, callType 을 다 봐야 함
create({ ..., int memoType = 0, int callType = 0, String color = '' });
```
이건 **인터페이스 분리 원칙(ISP) 위반** 신호다 — "안 쓰는 것에 의존하게 만든다".
실제로 기타용 `color` 를 추가했더니 주소증/메모 시그니처까지 바뀌었다(한 도메인 변경이 전체로 번짐).

해법: 변형 속성을 **값객체로 묶고, 도메인 팩토리로 의도를 드러낸다.**
```dart
class PatientMemoWrite {
  const PatientMemoWrite._({required this.category, this.memoType = 0, this.callType = 0, this.color = ''});
  final MemoCategory category; final int memoType; final int callType; final String color;
  const PatientMemoWrite.jusojung() : this._(category: MemoCategory.jusojung);          // 변형 없음
  const PatientMemoWrite.memo({int memoType = 0, int callType = 0})
      : this._(category: MemoCategory.memo, memoType: memoType, callType: callType);
  const PatientMemoWrite.etc({String color = ''}) : this._(category: MemoCategory.etc, color: color);
}

// create 시그니처는 깔끔해지고:
create({ ..., required PatientMemoWrite write, bool top = false });
// 호출측은 의도가 한눈에:
service.create(..., write: const PatientMemoWrite.jusojung());
service.create(..., write: PatientMemoWrite.etc(color: token));
```
효과: 주소증 호출자는 더 이상 `color` 를 안 본다. 새 변형이 생겨도 **공용 시그니처를 안 건드린다.**
파일·서비스는 여전히 1개라 "단일 엔드포인트 = 단일 서비스" 원칙도 유지.

## 7. 이름은 의미가 같으면 통일하라 — `isUnread` → `isNew`

`Etc` 만 N 배지 플래그를 `isUnread` 라 불렀는데, 메모/주소증은 같은 의미를 `isNew`("오늘 작성") 라 불렀다.
의미가 같은데 이름이 다르면 읽는 사람이 "뭔가 다른가?" 하고 멈칫한다. → `isNew` 로 통일.
(freezed 모델 필드를 바꿨으니 `build_runner` 재생성 필요: `flutter pub run build_runner build -d`)

## 8. 흔한 함정 / 헷갈리는 점

- **Mock 픽스처 ID 불일치**: 화면 코드가 멀쩡해도 "데이터가 안 뜬다." 도메인마다 `customerId` 가 다르면
  조회 키가 안 맞아 빈 결과가 난다. 픽스처는 **공통 환자 ID 규약**(예: 10001=김환자)을 전 도메인에서 지켜야 한다.
- **`family` 키를 상수로 박기**: 이번 버그처럼 `customerId` 를 상수로 두면 모든 입력이 같은 캐시를 본다.
  키는 반드시 입력(scheduleId→customerId)에서 도출.
- **DTO 를 화면에서 직접 쓰기**: 편해 보여도 서버 필드명 변화에 화면이 깨진다. Mapper 한 겹을 꼭 둔다.
- **build_runner 깜빡**: freezed/riverpod 애노테이션을 바꾸고 재생성을 안 하면 `_$...` 가 안 맞아 빨간 줄.
- **선택 파라미터 누적**: 편하다고 옵션 파라미터를 계속 추가하면 인터페이스가 오염된다. 3개 넘어가면 값객체를 의심.
